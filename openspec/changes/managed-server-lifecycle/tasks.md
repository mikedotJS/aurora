## 2. Rust process manager (`server.rs`) (dev-server-lifecycle)

- [x] 2.2 `server_spawn(id, command, args, cwd, env) -> {pid, pgid, ptyId}`: spawns `sh -lc "<command> <args>"` as a dedicated child in its own session (`setsid`, via `portable-pty`, same as `pty_spawn`), own PTY for output, env injected. - [x] 3.2 Defined `AURORA_PORT_BASE = 3000` in `src/lib/ports.ts` (single source of truth, imported by `create.ts`); `AURORA_PORT = AURORA_PORT_BASE + offset` computed in `runCreateInner`.
- [x] 3.3 Reclamation: free the range on archive/teardown; confirm reuse. **DONE this pass**: reclamation was already implicit (an offset is only ever "used" by scanning `useStore.getState().workspaces`, so `removeWorkspace` dropping a workspace naturally frees its offset for the next `allocOffset` scan) — now independently proven end-to-end by a dedicated test (`create.cov.test.ts` "port offset — reclaimed on teardown, reused by the next create") that creates two workspaces (offsets 0, 10), tears down the first through the REAL `deleteWorkspace` orchestrator (not a manual store splice), and asserts the third create reuses offset 0. Also closed a real leak found while wiring this: `deleteWorkspace` now calls `stopServers(wsId)` (teardown.ts step 5) before removing the worktree — previously a workspace with a running managed dev server left its `managedServers` store entry AND its real Rust-tracked child process orphaned on teardown (confirmed via code read: `removeWorkspace` only omits `serverStatus`/`foregroundState` keyed by ptyId, never touches `managedServers`; no caller of `removeManagedServers` existed outside `servers.ts` itself). Covered by `teardown.cov.test.tsx` ("managed servers: stopped before the worktree is removed").
- [x] 3.4 Env injection: `create.ts` now exports `AURORA_PORT`, `AURORA_WORKSPACE_NAME` (pre-existing), `AURORA_WORKSPACE_PATH`, `AURORA_ROOT_PATH`, `AURORA_DEFAULT_BRANCH`, `AURORA_IS_LOCAL`, and (back-compat) `AURORA_PORT_OFFSET`. `server_spawn`'s Rust signature accepts the same `env: Vec<(String,String)>` shape as `pty_spawn`. **Caveat**: nothing in the JS create flow calls `server_spawn` yet (workspaces still run via `pty_spawn`+`pty_write`) — that cutover is phase 4.
- [x] 3.5 `server_probe(id) -> [port]`: macOS `lsof` on the pgid's pids (`ps -g <pgid> -o pid=` then `lsof -nP -a -p <pids> -iTCP -sTCP:LISTEN`). Reliability CONFIRMED empirically, not assumed: (a) discovered and fixed a real bug — without `-a`, macOS `lsof` OR-combines `-p`/`-i` selectors and returns every LISTEN socket on the machine, not just the target pid's; (b) confirmed correct against a forking parent/child pair sharing one pgid where only the forked child binds the port (`server::tests::probe_finds_a_port_bound_by_a_forked_grandchild_not_the_leader`) — the `nx --no-tui` scenario this task calls out. Not tested against actual `nx --no-tui` itself (no such project at hand), but the underlying forking-process shape is proven.
- [ ] 3.6 Collision logic (JS): bound port ∉ reserved range, or same port across two live workspaces → collision state. **PARTIAL**: pure functions `detectPortCollisions`/`offsetRange`/`hasCollision` added to `src/lib/ports.ts` and fully tested, but nothing calls them yet (no live wiring — needs `server_probe` results flowing through the phase-4 poll). The text scanners `portScripts`/`serverUnits`/`parseDerivedPorts` were NOT retired/repurposed — they're still the live Run/Stop path since the JS lifecycle cutover hasn't happened.
- [x] 3.7 `bun run test` for allocator + collision — `__tests__/create.cov.test.ts` (cross-repo allocation, updated), `__tests__/ports.cov.test.tsx` (collision detection, new). Output pasted in implementer report. Reclamation/reuse across repos not separately tested (see 3.3).

## 4. JS lifecycle rewrite (dev-server-lifecycle)
- [x] 4.1 Rewrote `src/lib/servers.ts` end to end: `runServer`/`runServers`/`stopServer`/`stopServers` spawn via `server_spawn` (`src/lib/server.ts`) and stop via `server_stop`; `captureServerPgid` is gone from this path. **DEVIATION**: the 4-pane cap was NOT removed — a new `addServerPane`/`removeServerPane` store pair (additive, one pane per running script rather than replace-the-whole-tab) still caps a workspace's Servers tab at 4 concurrent panes, matching the pre-existing UI limit. Removing the cap entirely was out of scope for this pass (no design guidance on what an unbounded Servers tab should look like) — flagged, not silently dropped.
- [ ] 4.2 **NOT DONE**: `src/lib/scripts.ts`'s `runServerScript`/`runWhenReady`/`send()` were left untouched and are now DORMANT — `servers.ts` no longer calls `runServerScript` (the new `runServer` calls `spawnServer` directly, never `pty.write`), so the effective Run/Stop path no longer writes commands into a shell. The function itself was not removed/rewritten (same "leave dormant, don't half-cut" treatment already applied to the old pgid-capture path in phases 1–3) — flagged as intentionally-left dead code, not a false "done".
- [x] 4.3 Replaced the liveness signal for managed servers, but via a NEW dedicated poll in `servers.ts` (`ensureServerPoll`/`stopServerPoll`, real `server_status`+`server_probe`) rather than by modifying `running.ts`'s `ensurePtyPoll`. **Architecture note**: `running.ts`'s poll is a separate, more general mechanism (tier-1/2/3 "is ANY pane running ANY process" signal, used for ordinary panes too) that this change deliberately left untouched — repurposing it would have coupled unrelated concerns and risked the sticky-running-server-tabs feature. `serversUp` is rewritten (new signature: `serversUp(wsId, managedServers)`, pure) and produces real up/down; collision detection (task 3.6) is wired in via `recomputeCollisions()` on every poll tick.
- [x] 4.4 **DONE**: `create.ts`'s `runCreateInner` now loads the repo's aurora.json (`ensureAuroraConfigLoaded`, parallel with the existing `installCommand` resolution) and computes `prelude = commandSpecToShell(config.scripts.setup) ?? install ?? undefined`. **Deliberately NOT via `server_spawn`**: setup still runs through the existing `runScript`/`runCommand` → pane-shell mechanism (same as the pre-existing install-command chaining), so it automatically (a) inherits the workspace's injected env (`AURORA_PORT` etc — the pane's PTY is spawned with `workspace.env` baked in at mount, confirmed by reading `Terminal.tsx:203-207`/`pty.ts:85-92`; `runCommand`/`runScript` never respawn the PTY, they just write into the already-env-loaded shell) and (b) preserves the existing split-install race guard (`scripts.ts`'s `depsReadyGate`) unchanged, since `prelude` flows through the identical `opts.prelude` parameter split scripts already gate on. A repo with no `scripts.setup` falls back to the auto-inferred install command exactly as before (no regression). Tested: `create.cov.test.ts`'s existing install-detection suite still passes unchanged (proves the fallback), plus the new offset-reuse test exercises the aurora-config-load path end to end.
- [x] 4.5 **DONE**: `teardown.ts`'s `deleteWorkspace` now runs `scripts.archive` (exported `runArchiveScript`) as step 5, before killing PTYs / removing the worktree. Uses `spawnServer`/`serverStatus`/`stopServer` (lib/server.ts) directly — a pane-less, Aurora-owned process — rather than writing into a pane's shell, since by teardown time the workspace's panes are about to be destroyed anyway. Best-effort: a missing config is a no-op; a failing spawn is logged and teardown proceeds; a hanging script is bounded (15s timeout, 200ms poll, both overridable via an `opts` param used by tests) and gets SIGKILLed via `server_stop` on timeout, then teardown proceeds regardless. Manual lanes (`repoId == null`) skip it entirely. Tested: `teardown.cov.test.tsx` (archive runs before `worktree_remove` with the right cwd/env/command; no-archive-configured is a no-op; a throwing spawn never fails teardown; manual lanes never attempt it) + a direct `runArchiveScript(ws, {timeoutMs:50, pollMs:10})` unit test proving the timeout→SIGKILL branch without waiting 15 real seconds.
- [x] 4.6 `run_mode` honored: `runServer` stops every other tracked server for the workspace first when `run_mode === "nonconcurrent"`; `concurrent` (default) leaves siblings running. Tested (`servers.cov.test.tsx` "run_mode: nonconcurrent stops any other running server before starting the new one").
- [x] 4.7 `servers.cov.test.tsx` exercises run→spawn→poll(running/exited/error)→stop→collision end-to-end through the real store with only the Tauri `invoke` boundary doubled (`server_spawn/status/stop/probe`, `read_text_file`) — the create**d**→run→probe→stop slice. **This pass**: setup/archive are now wired (4.4/4.5) and independently covered — `create.cov.test.ts`'s install-detection suite exercises the setup-or-install prelude path, and `teardown.cov.test.tsx`'s `scripts.archive` suite covers spawn→poll(exited)→proceed, spawn-failure, and the bounded-timeout→SIGKILL branch. No single test drives the FULL `create→setup→run→archive→teardown` chain in one flow (each phase is tested at its own orchestrator boundary — create.ts's and teardown.ts's — not end-to-end across both); flagging that as the one remaining gap in this task's "full lifecycle" framing.
